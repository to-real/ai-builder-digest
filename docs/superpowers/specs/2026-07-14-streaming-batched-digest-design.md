# AI Builders Digest 流式分批生成设计

## 背景

当前 `remix.mjs` 把当天全部 X、博客和播客内容连同完整提示词一次性交给 GLM，并等待模型一次性返回整篇双语 Markdown。2026-07-14 的公开 feed 约为 114 KB，其中播客约 85 KB。真实 Actions 验证中，两个 GLM 请求分别在 300007 ms 和 300003 ms 后被客户端超时中止。

当前做法还有提示词重复问题：`prepare-digest.js` 把远程 prompts 放进 `feed.json`，`remix.mjs` 又把仓库内 prompts 拼进 system message。重复内容增加输入体积，也可能造成指令冲突。

## 目标

- 每天尽量发布一份可读日报，单个异常条目不能阻塞整期内容。
- 最终编辑器仍能看到所有成功摘要，从而完成跨来源主题合并、排序和文风统一。
- 通过流式响应区分“正在生成”和“连接无进展”，不再只使用固定的整体请求超时。
- 支持失败后续跑；已完成的条目不重复调用 GLM。
- 最终输出继续满足现有双语 Markdown、来源链接和页面布局规则。

## 非目标

- 不更换 GLM 模型或供应商。
- 不改变静态站点的页面设计。
- 不保证每个原始条目都必须进入日报；连续失败的单条内容可以跳过。
- 第一版不根据 API 价格动态选择模型或并发数。

## 总体方案

采用“分批结构化压缩 + 全局流式编辑 + 确定性模板兜底”的两阶段架构。

```text
原始 feed
  -> 移除重复 prompts 并标准化条目
  -> 按输入体积分批
  -> 流式生成结构化摘要 JSON
  -> 校验并缓存每条摘要
  -> 全局编辑器读取全部成功摘要
  -> 流式生成双语 Markdown
  -> 格式校验
  -> 正常发布，或使用确定性模板兜底
```

仓库内 `prompts/*.md` 是唯一的生产提示词来源。它们受版本控制，便于复现输出和计算缓存版本。`feed.json` 不再包含远程 prompts。

## 输入标准化与分批

推文、博客和播客统一转换为内容条目：

```json
{
  "id": "<type-and-url-hash>",
  "type": "x | blog | podcast",
  "author": "...",
  "bio": "...",
  "title": "...",
  "content": "...",
  "url": "...",
  "publishedAt": "..."
}
```

按序向批次中加入条目，每批输入目标上限为 20,000 字符。批次上限按实际序列化后的字符数计算，而不是固定条目数；短推文可以同批处理，长播客会自动占用更少条目。单条内容超过上限时允许独立成批，不能静默截断原文。

同时最多执行两个 GLM 批次。该并发数优先保障稳定性，后续只有在真实运行数据证明不存在限流和排队问题时才提高。

## 第一阶段：结构化摘要

批次请求只使用对应内容类型的摘要提示词，不负责最终 Markdown 排版。每个内容条目输出：

```json
{
  "id": "...",
  "type": "x | blog | podcast",
  "author": "...",
  "title": "...",
  "include": true,
  "role": "...",
  "judgment": "...",
  "facts": ["..."],
  "english": "...",
  "chinese": "...",
  "themes": ["..."],
  "url": "..."
}
```

响应采用流式传输，但必须在流结束后拼接、解析并按 schema 校验。缺少 `id`、`type`、作者、必要标题、角色、`url`、核心判断或双语素材的记录视为无效，不能进入最终编辑阶段。

每个有效条目立即写入 `.digest-cache/items/<cache-key>.json`。缓存键由规范化内容哈希、内容类型、模型名和相关提示词哈希共同生成。内容或提示词变化时缓存自然失效。

GitHub Actions 在摘要阶段之前恢复缓存，并在摘要阶段完成后立即保存缓存；最终编辑失败不影响已经完成的摘要在手动重跑时被复用。

## 第二阶段：全局编辑

全局编辑器只接收成功且已校验的结构化摘要，不再读取完整 transcript 或原始推文。它负责：

- 识别跨来源的共同主题并合并重复观点。
- 按重要性排序，过滤过薄内容。
- 统一中英文信息密度、判断句写法和作者角色格式。
- 生成符合现有站点布局要求的完整 Markdown。

该请求同样使用流式响应。流结束后执行本地格式校验，包括：

- 一级标题、栏目标题和条目标题层级正确。
- 每个条目同时包含英文和中文内容。
- 每个收录条目至少包含一个来自结构化摘要的原始 URL。
- 不得出现结构化摘要集合之外的新 URL。
- 结尾保留 Follow Builders 声明。

格式校验失败时，将精确错误和上一版 Markdown 交给编辑器修复一次，而不是重新运行第一阶段。

## 超时、重试与降级

流式请求使用三类时间限制：

- 首个数据块超时：120 秒。
- 流式空闲超时：连续 60 秒没有收到新数据。
- 绝对时长：摘要批次 5 分钟，最终编辑 8 分钟。

每个摘要批次失败后等待并重试一次。第二次仍失败时，将该批次按输入体积拆成两个更小批次并继续处理。拆分递归到单条内容；单条仍失败则跳过，记录原因，但不使整期日报失败。

最终编辑或格式修复仍失败时，本地确定性渲染器直接使用有效结构化摘要生成基础双语 Markdown。兜底版本不进行复杂的跨条目主题合并，但必须保持合法标题层级、双语段落和来源链接，确保站点可以构建和发布。

如果当天没有任何有效摘要，则工作流失败，不发布空日报。

## 可观测性

每次运行生成一份机器可读报告，至少包含：

- feed 总体积和各内容类型体积。
- 批次数、每批条目数和输入字符数。
- 缓存命中数。
- 每次 GLM 调用的首个数据块耗时、总耗时、数据块数量和结束原因。
- 重试、批次拆分和跳过条目数量。
- 最终使用正常编辑、格式修复还是确定性兜底。

日志只记录条目 ID、URL 和统计信息，不记录 API 密钥或完整原始正文。运行报告在失败时作为 Actions artifact 上传，便于诊断。

## 组件边界

- `prepare-digest.js`：获取 feed，只输出内容与统计数据，不携带 prompts。
- `normalize-feed.mjs`：把不同来源转换为标准条目并生成稳定 ID。
- `batch-items.mjs`：按序列化体积打包和拆分批次。
- `glm-stream.mjs`：负责 SSE 解析、三类超时、错误分类和调用指标。
- `summarize-items.mjs`：批次摘要、schema 校验、重试和缓存。
- `compose-digest.mjs`：全局编辑、Markdown 校验和一次格式修复。
- `render-fallback.mjs`：不调用模型的确定性兜底渲染。
- `remix.mjs`：只负责组织上述阶段和写入最终 digest。

这些模块通过标准条目、结构化摘要和调用结果对象通信，避免测试代码依赖真实 GLM。

## 测试策略

单元测试覆盖：

- 不同长度条目的分批边界和顺序稳定性。
- 超大单条内容独立成批。
- 缓存键随正文、模型或提示词变化而变化。
- SSE 数据块解析、结束标记和不完整流。
- 首数据块、空闲和绝对超时。
- 批次重试、二分拆分及单条跳过。
- 结构化摘要 schema 校验。
- Markdown 标题、双语段落和 URL 白名单校验。
- 确定性兜底渲染器输出合法站点输入。

集成测试使用本地假 GLM 服务模拟：正常流、延迟首包、流中停顿、连接中断、无效 JSON、格式错误 Markdown 和部分批次永久失败。测试不得等待真实分钟数，所有时间通过可注入时钟控制。

GitHub Actions 验收使用固定 fixture 先验证完整流水线，再通过 `workflow_dispatch` 执行一次真实 feed。验收标准是工作流成功发布；真实 GLM 中单条永久失败可以被跳过，但必须在运行报告中可见。

## 发布与回滚

第一版保持现有单次生成代码作为短期回滚路径，通过工作流环境变量选择新旧流水线。新流水线至少连续成功运行三次后，再删除旧路径和开关。

如果新流水线出现严重质量问题，可立即切回旧路径；缓存和结构化摘要均属于可删除的派生数据，不影响历史 `digests/*.md`。
